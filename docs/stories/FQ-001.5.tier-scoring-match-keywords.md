# Story FQ-001.5: Refatoracao match_keywords com Tier Scoring

**Epic:** FQ-001 — Eliminacao de Falsos Positivos
**Status:** Draft
**Story Points:** 8
**Prioridade:** P1 — Alta
**Dependencia:** FQ-001.1, FQ-001.2, FQ-001.3, FQ-001.4

---

## Story

**As a** mantenedor do Descomplicita,
**I want** que o sistema de matching use pesos por tier de keyword ao inves de match binario,
**so that** keywords ambiguos sozinhos nao aprovem itens e a taxa de FP caia estruturalmente para < 1%.

---

## Story Context

**Existing System Integration:**
- Integrates with: `backend/filter.py` (funcao `match_keywords`)
- Integrates with: `backend/sectors.py` (SectorConfig.keywords)
- Technology: Python, regex word boundary matching
- Touch points: `filter.py`, `sectors.py`, `main.py`, `tests/test_pncp_client.py`

**Problema Arquitetural:**
O `match_keywords()` atual retorna `True` se QUALQUER keyword matcha (match binario). Keywords ambiguos como "servidor", "infraestrutura", "EPI", "limpeza", "cobertura" isolados geram FPs porque nao exigem contexto.

As exclusoes (Fase 1) resolvem padroes conhecidos, mas novos padroes surgirao. O tier scoring e a solucao estrutural.

---

## Acceptance Criteria

### Functional Requirements

1. `SectorConfig` aceita keywords organizados por tier: `keywords_a` (inequivocos), `keywords_b` (fortes), `keywords_c` (ambiguos)
2. Backward compatibility: `keywords` (Set flat) continua funcionando como atalho (todos tier A)
3. `match_keywords()` calcula score ponderado: A=1.0, B=0.7, C=0.3
4. Cada setor define um `threshold` (padrao 0.6)
5. Item e aprovado quando `score >= threshold`
6. Keywords tier C sozinhos (score 0.3) nao aprovam item

### Integration Requirements

7. Setores que NAO usam tiers (keywords flat) continuam funcionando identicamente
8. `filter_licitacao()` e `filter_batch()` nao precisam de alteracao
9. API `/buscar` nao precisa de alteracao
10. Frontend nao precisa de alteracao

### Quality Requirements

11. Cobertura de testes >= 90% para nova logica de scoring
12. Todos os testes existentes continuam passando
13. Auditoria com >= 3000 itens mostra FP < 1% nos 5 setores problematicos
14. Zero regressao: setores que tinham 0% FP continuam com 0%

## Design Tecnico

### Modelo de Dados

```python
@dataclass(frozen=True)
class SectorConfig:
    id: str
    name: str
    description: str
    keywords: Set[str]           # Mantido para backward compat (= tier A)
    exclusions: Set[str] = field(default_factory=set)
    # Novo: keyword tiers (opcional)
    keywords_a: Set[str] = field(default_factory=set)  # Inequivocos (peso 1.0)
    keywords_b: Set[str] = field(default_factory=set)  # Fortes (peso 0.7)
    keywords_c: Set[str] = field(default_factory=set)  # Ambiguos (peso 0.3)
    threshold: float = 0.6                              # Score minimo
```

**Regra de resolucao:**
- Se `keywords_a` esta preenchido → usar tiers (keywords_a + _b + _c)
- Se `keywords_a` esta vazio → usar `keywords` como tier A (backward compat)
