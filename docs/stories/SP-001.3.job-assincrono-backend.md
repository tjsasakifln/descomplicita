# Story SP-001.3: Job Assincrono — Backend Async com Polling

**Epic:** SP-001 — Search Pipeline Performance
**Status:** Draft
**Story Points:** 8
**Prioridade:** P1 — Alta

---

## Story

**As a** usuario do BidIQ,
**I want** que a busca inicie imediatamente e me mostre progresso em tempo real,
**so that** eu saiba que o sistema esta trabalhando e nunca veja um erro de timeout.

---

## Story Context

**Existing System Integration:**
- Integrates with: `backend/main.py` (POST /buscar), `backend/pncp_client.py` (fetch_all)
- Technology: FastAPI BackgroundTasks ou asyncio.create_task
- Touch points: `main.py` (novo endpoint pattern), `pncp_client.py` (callbacks de progresso)

**Problema Atual:**
O endpoint `POST /buscar` e sincrono — faz fetch, filtra, gera LLM, gera Excel, e so retorna quando tudo termina. Se o processo demora mais de 5-8min, o frontend aborta a conexao.

**Solucao:**
Transformar em **job assincrono**: POST cria o job e retorna imediatamente, GET consulta status/progresso, GET final retorna resultado.

**Dependencia:** SP-001.1 (quick wins ja aplicados reduzem latencia base)

---

## Acceptance Criteria

### Job Creation

1. `POST /buscar` retorna em < 1s com `{ job_id, status: "queued" }`
2. Job ID e UUID v4 unico
3. Processamento inicia em background task imediatamente apos retorno
4. Request body identico ao atual (zero breaking change no contrato)

### Job Status & Progress

5. `GET /buscar/{job_id}/status` retorna estado atual do job:
   ```json
   {
     "job_id": "uuid",
     "status": "running|completed|failed",
     "progress": {
       "phase": "fetching|filtering|summarizing|generating_excel|done",
       "ufs_completed": 2,
       "ufs_total": 5,
       "items_fetched": 342,
       "items_filtered": 28
     },
     "created_at": "2026-03-06T10:00:00Z",
     "elapsed_seconds": 45
   }
   ```
6. Progresso atualizado a cada UF completada durante fase de fetch
7. Progresso atualizado em cada transicao de fase (fetch -> filter -> LLM -> Excel)

### Job Result

8. `GET /buscar/{job_id}/result` retorna resultado completo quando `status: "completed"`
9. Formato de resposta identico ao atual `POST /buscar` (resumo, download_id, totais, filter_stats)
10. Se job ainda nao completou, retorna `{ status: "running" }` com HTTP 202
11. Se job falhou, retorna `{ status: "failed", error: "mensagem" }` com HTTP 500

### Job Lifecycle

12. Jobs expiram apos 30 minutos (cleanup automatico)
13. Maximo de 10 jobs simultaneos (rejeitar novos com HTTP 429 se exceder)
14. Job store e dicionario em memoria (sem dependencia externa)
15. Se backend reinicia, jobs em andamento sao perdidos (aceitavel para v1)

### Backward Compatibility

16. Frontend proxy `POST /api/buscar` em `route.ts` atualizado para novo fluxo (cria job + poll)
17. Endpoint antigo sincrono removido (substituido pelo novo)

### Quality Requirements

18. Testes para: criacao de job, polling de status, resultado completo, job expirado, limite de jobs
19. Nenhum job "orphan" apos 30 minutos (cleanup verificado)

## Technical Notes

### Arquitetura do Job Store

```python
# Em main.py ou novo modulo job_store.py
@dataclass
class SearchJob:
    job_id: str
    status: Literal["queued", "running", "completed", "failed"]
    progress: dict
    result: dict | None
    error: str | None
    created_at: float
    completed_at: float | None

class JobStore:
    def __init__(self, max_jobs=10, ttl=1800):
        self._jobs: dict[str, SearchJob] = {}
        self._lock = threading.Lock()

    def create(self, job_id: str) -> SearchJob: ...
    def update_progress(self, job_id: str, **kwargs): ...
    def complete(self, job_id: str, result: dict): ...
    def fail(self, job_id: str, error: str): ...
    def get(self, job_id: str) -> SearchJob | None: ...
    def cleanup_expired(self): ...
```

### Fluxo de Execucao

```
1. POST /buscar
   -> Cria SearchJob(status="queued")
   -> asyncio.create_task(run_search(job))
   -> Retorna { job_id }

2. run_search(job):
   -> job.status = "running"
   -> fetch_all() com callback de progresso
   -> filter_batch()
   -> gerar_resumo_llm()  |  em paralelo
   -> gerar_excel()        |
   -> job.status = "completed", job.result = {...}

3. GET /buscar/{job_id}/status
   -> Retorna job.progress (polling a cada 2-3s pelo frontend)

4. GET /buscar/{job_id}/result
   -> Retorna job.result quando completed
```

### Callback de Progresso no pncp_client

Adicionar parametro `on_progress` ao `fetch_all()`:

```python
def fetch_all(self, ..., on_progress: Callable | None = None):
    # Apos cada UF completar:
    if on_progress:
        on_progress(uf=uf, items=len(results))
```
