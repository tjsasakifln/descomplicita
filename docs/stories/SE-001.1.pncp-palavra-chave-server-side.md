# Story SE-001.1: Implementar Filtro `palavraChave` Server-Side no PNCP

**Epic:** SE-001 — Search Effectiveness
**Status:** Ready
**Story Points:** 5
**Prioridade:** P0 — Critica

---

## Story

**As a** usuario buscando licitacoes de um setor especifico,
**I want** que a busca no PNCP use filtro server-side por palavra-chave,
**so that** eu receba apenas licitacoes relevantes ao meu setor em vez de zero resultados.

---

## Story Context

**Existing System Integration:**
- Arquivos: `backend/pncp_client.py`, `backend/sources/pncp_source.py`, `backend/sources/base.py`, `backend/main.py`
- Testes: `backend/tests/test_pncp_client.py`, `backend/tests/test_orchestrator.py`
- Dependencia: nenhuma

**Problema:**
A API PNCP (`/contratacoes/publicacao`) aceita o parametro `palavraChave` que busca dentro do campo `objetoCompra` server-side. Atualmente, `pncp_client.py:243-252` NAO envia este parametro. O sistema busca TODAS as licitacoes de TODOS os setores e depois filtra client-side com keywords, resultando em match rate proximo a 0% para setores especificos em janelas curtas de tempo.

**Solucao:**
Propagar keywords prioritarias do setor (`SectorConfig.search_keywords`) atraves do pipeline:
`main.py` -> `SearchQuery` -> `orchestrator` -> `PNCPSource` -> `PNCPClient.fetch_all()` -> API PNCP (`palavraChave=X`)

Para cada setor, selecionar 3-5 termos de busca de alta precisao (ex: "medicamento", "hospitalar", "farmaceutico" para saude) e executar uma query PNCP para cada termo. Isto reduz drasticamente o volume de dados irrelevantes e aumenta a taxa de match.

---

## Acceptance Criteria

### 1. SearchQuery aceita keywords de busca

- `SearchQuery` (em `sources/base.py`) inclui campo opcional `search_terms: Optional[List[str]]`
- Campo propagado pelo orchestrator ate cada source

### 2. SectorConfig define termos de busca prioritarios

- Novo campo `search_keywords: List[str]` em `SectorConfig` (em `sectors.py`)
- Maximo 5 termos por setor (para limitar queries)
- Termos de alta precisao, NAO a lista completa de keywords
- Exemplo para saude: `["medicamento", "hospitalar", "farmaceutico", "insumo hospitalar", "material hospitalar"]`

### 3. PNCPClient envia `palavraChave` na API

- `pncp_client.py:fetch_page()` aceita parametro opcional `palavra_chave: str`
- Quando presente, adiciona `params["palavraChave"] = palavra_chave`
- `fetch_all()` aceita `search_terms: List[str]` e executa N buscas independentes (uma por termo)
- Deduplicacao por `numeroControlePNCP` entre resultados de diferentes termos

### 4. PNCPSource propaga search_terms

- `pncp_source.py:fetch_records()` passa `query.search_terms` para `PNCPClient.fetch_all()`
- Se `search_terms` vazio/None, comportamento identico ao atual (busca sem filtro)

### 5. main.py envia search_keywords do setor

- `run_search_job()` adiciona `sector.search_keywords` ao `SearchQuery`
- Custom terms (`request.termos_busca`) substituem search_keywords quando presentes

### 6. Backward compatibility

- Sem `search_terms`, pipeline funciona identicamente ao atual
- Filtro client-side (`filter_batch`) continua aplicando keywords completas apos fetch
- Testes existentes continuam passando sem modificacao

### 7. Performance

- Busca com `palavraChave` retorna menos dados => pipeline mais rapido
- Tempo total nao deve exceder 120s para 27 UFs x 5 modalidades x 3 termos

## Tarefas Tecnicas

1. Adicionar `search_terms: Optional[List[str]]` a `SearchQuery` em `sources/base.py`
2. Adicionar `search_keywords: List[str]` a `SectorConfig` em `sectors.py`
3. Definir `search_keywords` para todos os 12 setores
4. Modificar `PNCPClient.fetch_page()` para aceitar `palavra_chave`
5. Modificar `PNCPClient.fetch_all()` para iterar sobre `search_terms`
6. Modificar `PNCPSource.fetch_records()` para propagar `search_terms`
7. Modificar `run_search_job()` em `main.py` para passar `sector.search_keywords`
8. Adicionar testes unitarios para cada camada modificada
9. Teste de integracao: busca "saude" com palavraChave retorna > 0 resultados

## Riscos

| Risco | Mitigacao |
|-------|----------|
| `palavraChave` PNCP pode ter comportamento diferente | Testar manualmente via curl antes de implementar |
| N termos x M UFs x K modalidades = explosao combinatoria | Limitar a 3-5 termos; paralelizar com ThreadPoolExecutor |
| Rate limiting PNCP (10 RPS) pode throttle | Respeitar rate limit existente; termos sao sequenciais por UF |
