# Story SR-001.1: Corrigir PNCP Orchestrator Timeout (15s para 120s)

**Epic:** SR-001 — Search Resilience
**Status:** Ready
**Story Points:** 3
**Prioridade:** P0 — Critica

---

## Story

**As a** usuario buscando licitacoes,
**I want** que a busca no PNCP tenha tempo suficiente para completar,
**so that** eu receba os resultados disponiveis em vez de "nenhuma licitacao encontrada".

---

## Story Context

**Existing System Integration:**
- Arquivos: `backend/config.py`, `backend/sources/orchestrator.py`
- Testes: `backend/tests/test_orchestrator.py`, `backend/tests/test_sources_config.py`
- Dependencia: nenhuma

**Problema:**
O `SOURCES_CONFIG["pncp"]["timeout"]` em `config.py:75` esta configurado em 15 segundos. Este valor e usado pelo orchestrator (`orchestrator.py:211-213`) como timeout para **toda a operacao de busca** do PNCP (7 UFs x 5 modalidades = 35 combinacoes paralelas, potencialmente milhares de paginas).

A operacao real leva 2-5 minutos. O timeout de 15s coincide com o `RetryConfig.timeout` (timeout HTTP por request individual em `config.py:49`), indicando confusao entre os dois conceitos.

**Resource Leak:** Quando `asyncio.wait_for()` faz timeout, as threads do `ThreadPoolExecutor` em `pncp_client.py:485` continuam executando em background — confirmado nos logs Railway.

---

## Acceptance Criteria

### Timeouts corrigidos

1. `SOURCES_CONFIG["pncp"]["timeout"]` alterado de 15 para 120 (segundos)
2. `SOURCES_CONFIG["comprasgov"]["timeout"]` alterado de 20 para 60
3. `SOURCES_CONFIG["transparencia"]["timeout"]` alterado de 30 para 90
4. `SOURCES_CONFIG["querido_diario"]["timeout"]` alterado de 20 para 60
5. `SOURCES_CONFIG["tce_rj"]["timeout"]` alterado de 30 para 90

### Clareza no codigo

6. Comentario inline em `SOURCES_CONFIG` explicando que `timeout` se refere ao orchestrator (operacao completa), nao ao timeout HTTP individual
7. Comentario em `RetryConfig.timeout` referenciando que este e o timeout HTTP por request

### Validacao

8. Busca com 7 UFs x 5 modalidades completa sem timeout do orchestrator
9. Testes existentes em `test_orchestrator.py` e `test_sources_config.py` atualizados
10. Todos os testes passando

### Uso do timeout no orchestrator

```python
# orchestrator.py:211-213
timeout = SOURCES_CONFIG.get(
    source.source_name, {}
).get("timeout", self.default_timeout)

# orchestrator.py:303-305
records = await asyncio.wait_for(
    source.fetch_records(query),
    timeout=timeout,
)
```

### Nota sobre resource leak

O `asyncio.wait_for()` cancela o task async, mas `run_in_executor()` em `pncp_source.py:97` executa em thread separada que nao e cancelavel. Aumentar o timeout mitiga o problema (thread termina normalmente), mas nao resolve o leak fundamentalmente. Migracao para `httpx` async e recomendada como futuro epic.
