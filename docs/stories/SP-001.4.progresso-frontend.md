# Story SP-001.4: Progresso no Frontend — Barra de Progresso e Resultados Parciais

**Epic:** SP-001 — Search Pipeline Performance
**Status:** Draft
**Story Points:** 5
**Prioridade:** P1 — Alta

---

## Story

**As a** usuario do BidIQ,
**I want** ver uma barra de progresso mostrando quantos estados ja foram processados,
**so that** eu saiba que a busca esta funcionando e quanto falta para completar.

---

## Story Context

**Existing System Integration:**
- Integrates with: `frontend/app/page.tsx` (search flow), `frontend/app/api/buscar/route.ts` (proxy)
- Technology: Next.js 14, React state, fetch polling
- Touch points: `page.tsx` (UI de progresso), `route.ts` (polling logic)

**Dependencia:** SP-001.3 (backend async com endpoints de status/polling)

**Estado Atual do Frontend:**
- Ao buscar, exibe estado `loading` com texto generico
- Nenhum feedback de progresso durante a espera
- Timeout de 5min (sera 8min com SP-001.1) aborta silenciosamente

---

## Acceptance Criteria

### Polling Logic no Proxy

1. `POST /api/buscar` no frontend cria job no backend e inicia polling automatico
2. Polling a cada 2 segundos via `GET /buscar/{job_id}/status`
3. Polling para quando status e `completed` ou `failed`
4. Timeout de polling: 10 minutos (margem generosa, job backend expira em 30min)

### Barra de Progresso

5. Componente de progresso exibe: fase atual ("Buscando licitacoes...", "Filtrando resultados...", "Gerando resumo...", "Preparando download...")
6. Barra de progresso visual mostra `ufs_completed / ufs_total` durante fase de fetch
7. Contador de itens: "342 licitacoes encontradas ate agora" (atualiza em tempo real)
8. Animacao suave na barra (transition CSS, nao pulos bruscos)

### Estados da UI

9. **Queued:** "Iniciando busca..." (spinner)
10. **Fetching:** Barra de progresso + "Buscando em SP... (2/5 estados)" + contador de itens
11. **Filtering:** "Filtrando resultados..." (spinner, fase rapida)
12. **Summarizing:** "Gerando resumo inteligente..." (spinner)
13. **Generating Excel:** "Preparando planilha..." (spinner)
14. **Completed:** Transicao automatica para tela de resultados (comportamento atual)
15. **Failed:** Mensagem de erro amigavel com botao "Tentar novamente"

### Resultados Parciais (Nice-to-Have)

16. Se implementavel sem complexidade excessiva: mostrar contagem parcial de itens filtrados durante a fase de fetch (atualiza a cada UF completada)

### Remocao do AbortController

17. AbortController de timeout removido de `route.ts` (nao e mais necessario — polling controla lifecycle)
18. Cancelamento manual: botao "Cancelar busca" para o polling e exibe estado anterior

### Quality Requirements

19. Barra de progresso funciona em mobile (responsiva)
20. Transicoes entre fases sao suaves (sem flicker)
21. Nenhum memory leak no polling (clearInterval no cleanup)

## Technical Notes

### Polling no Proxy (route.ts)

```typescript
// Novo fluxo em route.ts
// 1. POST /buscar no backend -> { job_id }
// 2. Retornar job_id para page.tsx
// OU
// Alternativa: page.tsx faz polling diretamente
```

**Decisao recomendada:** `page.tsx` faz polling diretamente ao backend (via route proxy) para simplificar. Route proxy so repassa requests.

### State Machine no Frontend

```typescript
type SearchPhase =
  | "idle"
  | "queued"
  | "fetching"
  | "filtering"
  | "summarizing"
  | "generating_excel"
  | "completed"
  | "failed";

interface SearchProgress {
  phase: SearchPhase;
  ufs_completed: number;
  ufs_total: number;
  items_fetched: number;
  items_filtered: number;
  elapsed_seconds: number;
}
```

### Componente ProgressBar

```tsx
// Sugestao de estrutura
<SearchProgress
  phase={progress.phase}
  ufsCompleted={progress.ufs_completed}
  ufsTotal={progress.ufs_total}
  itemsFetched={progress.items_fetched}
/>
```
