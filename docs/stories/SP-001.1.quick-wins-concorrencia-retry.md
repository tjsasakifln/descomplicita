# Story SP-001.1: Quick Wins — Otimizacao de Concorrencia e Retry

**Epic:** SP-001 — Search Pipeline Performance
**Status:** Done
**Story Points:** 3
**Prioridade:** P0 — Critica

---

## Story

**As a** usuario do BidIQ buscando licitacoes em multiplos estados,
**I want** que a busca seja mais rapida e resiliente a falhas do PNCP,
**so that** eu consiga resultados sem timeout mesmo com 3+ UFs selecionados.

---

## Story Context

**Existing System Integration:**
- Integrates with: `backend/config.py` (RetryConfig), `backend/pncp_client.py` (fetch_all, ThreadPoolExecutor)
- Technology: Python ThreadPoolExecutor, httpx
- Touch points: `config.py`, `pncp_client.py`, `frontend/app/api/buscar/route.ts`

**Problema Atual:**
- `max_workers=6` limita paralelismo (15+ tasks ficam enfileiradas)
- `max_retries=5` com backoff ate 60s consome 62s no pior caso
- Frontend timeout de 5min e apertado para cenarios com retries
- LLM e Excel sao gerados sequencialmente apos todo o fetch

---

## Acceptance Criteria

### Otimizacao de Workers

1. `max_workers` em `pncp_client.py:fetch_all()` aumentado de 6 para 12
2. Monitoramento de respostas 429 (rate limit) — se > 20% dos requests retornam 429, log warning

### Otimizacao de Retry

3. `max_retries` em `config.py` reduzido de 5 para 3
4. `max_delay` em `config.py` reduzido de 60s para 15s
5. `base_delay` mantido em 2.0s (backoff total maximo: 2+4+8 = 14s vs 62s atual)

### Paralelizacao de Pos-Processamento

6. Geracao de resumo LLM e Excel executadas em paralelo (nao sequencialmente) em `main.py`
7. Tempo de pos-processamento reduzido de ~18s para ~15s (limitado pelo LLM)

### Frontend Timeout

8. Timeout do AbortController em `route.ts` aumentado de 5min para 8min (margem de seguranca)
9. Mensagem de timeout atualizada para refletir novo limite

### Quality Requirements

10. Todos os testes existentes continuam passando
11. Busca com 5 UFs + 7 dias completa sem timeout em ambiente de producao

---

## Technical Notes

### Arquivos a Modificar

| Arquivo | Mudanca |
|---------|---------|
| `backend/config.py` | `max_retries: 3`, `max_delay: 15.0` |
| `backend/pncp_client.py` | `max_workers=12` em ThreadPoolExecutor |
| `backend/main.py` | `asyncio.gather()` para LLM + Excel em paralelo |
| `frontend/app/api/buscar/route.ts` | Timeout de 5min para 8min |

### Impacto Esperado

| Metrica | Antes | Depois |
|---------|-------|--------|
| Max backoff por thread | 62s | 14s |
| Batches para 15 tasks | 3 (6 workers) | 2 (12 workers) |
| Pos-processamento | ~18s sequencial | ~15s paralelo |
| Frontend timeout | 300s | 480s |
| **Reducao estimada** | — | **~40-60% menos timeouts** |
