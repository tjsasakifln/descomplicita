# Story SP-001.2: Cache PNCP — Resultados em Memoria com TTL

**Epic:** SP-001 — Search Pipeline Performance
**Status:** Draft
**Story Points:** 5
**Prioridade:** P0 — Critica

---

## Story

**As a** usuario do BidIQ que busca licitacoes mais de uma vez por dia,
**I want** que buscas repetidas retornem instantaneamente,
**so that** eu nao espere minutos toda vez que refino minha pesquisa ou mudo de setor.

---

## Story Context

**Existing System Integration:**
- Integrates with: `backend/pncp_client.py` (fetch_all), `backend/main.py` (POST /buscar)
- Technology: Python dict com TTL, threading.Lock
- Touch points: `pncp_client.py` (cache layer), `main.py` (cache hit path)

**Insight Chave:**
O PNCP publica dados de licitacoes com frequencia diaria. Para a mesma combinacao de UF + modalidade + data, os resultados sao identicos durante o dia. Isso significa que 90%+ das buscas repetidas podem ser servidas do cache.

**Dados de Uso:**
- Usuarios tipicos fazem 3-5 buscas/dia variando setor e UFs
- Fetch PNCP e identico para mesmo UF+data independente do setor
- Filtragem por setor e local e instantanea (< 1s)

---

## Acceptance Criteria

### Cache Layer

1. Cache implementado como dicionario em memoria no `PNCPClient` com chave `f"{uf}:{modalidade}:{data_inicial}:{data_final}"`
2. TTL configuravel, default 4 horas (dados PNCP atualizam ~1x/dia)
3. Cache e thread-safe (usa `threading.Lock` ou `threading.RLock`)
4. Eviction automatico de entradas expiradas em cada acesso

### Cache Hit Path

5. Quando cache hit, `fetch_all()` retorna dados cacheados sem request HTTP
6. Log de nivel INFO registra "Cache hit for {uf}:{modalidade}" para monitoramento
7. Tempo de resposta para cache hit: < 100ms para a etapa de fetch

### Cache Miss Path

8. Quando cache miss, comportamento identico ao atual (fetch + armazena no cache)
9. Dados brutos (pre-filtragem) sao cacheados — filtragem por setor e aplicada apos

### Cache Management

10. Endpoint `GET /cache/stats` retorna: total de entradas, hits, misses, hit ratio
11. Endpoint `POST /cache/clear` limpa cache manualmente (para debug/deploy)
12. Cache size limitado a 500 entradas (LRU eviction se exceder)

### Quality Requirements

13. Testes unitarios para: cache hit, cache miss, TTL expiration, thread safety, LRU eviction
14. Busca repetida com mesmos UFs+datas retorna em < 5s (vs 40-120s sem cache)

## Technical Notes

### Design do Cache

```python
# Em pncp_client.py
@dataclass
class CacheEntry:
    data: list[dict]
    created_at: float  # time.time()
    ttl: float  # seconds

class PNCPClient:
    def __init__(self, ...):
        self._cache: dict[str, CacheEntry] = {}
        self._cache_lock = threading.Lock()
        self._cache_hits = 0
        self._cache_misses = 0
        self.cache_ttl = 4 * 3600  # 4 hours
        self.max_cache_entries = 500
```

### Cache Key Strategy

Cache e por **UF + modalidade + range de datas** (dados brutos do PNCP). A filtragem por setor/keywords e aplicada depois, permitindo que buscas de setores diferentes reusem o mesmo cache.

### Estimativa de Memoria

- ~1000 licitacoes por UF/modalidade/semana
- ~2KB por licitacao (JSON)
- 500 entradas x 2MB = ~1GB max (aceitavel para servidor)
